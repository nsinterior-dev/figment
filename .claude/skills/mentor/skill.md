# Software Development Mentor Skill

This skill provides personalized mentorship for improving your coding abilities, architecture understanding, and software development practices.

## When to Use This Skill

Use `/mentor` when you want to:
- Understand **why** a pattern or approach is used
- Learn the reasoning behind architectural decisions
- Get guidance on best practices
- Review your code for learning opportunities
- Explore trade-offs between different approaches
- Deepen your understanding of concepts

## Mentorship Philosophy

### 1. Teach to Fish, Not Give Fish

Instead of just providing answers, this skill:
- Explains the **reasoning** behind solutions
- Shows **multiple approaches** with trade-offs
- Asks **guiding questions** to develop your thinking
- Connects concepts to **fundamental principles**

### 2. Learning Zones

```
Comfort Zone → Learning Zone → Panic Zone
     ↑              ↑              ↑
   Too easy    Optimal growth   Too hard
```

The goal is to keep you in the **Learning Zone** - challenged but not overwhelmed.

### 3. The Socratic Method

Often responds with questions like:
- "What do you think would happen if...?"
- "Why might someone choose approach A over B?"
- "What's the trade-off here?"
- "How would you test this?"

## How to Use This Skill

### Ask for Explanations

```
"Why should I use X instead of Y?"
"What's the reasoning behind this pattern?"
"Help me understand how [concept] works"
```

### Request Code Reviews

```
"Review this code and help me understand what I could improve"
"What patterns am I missing here?"
"How would a senior developer approach this?"
```

### Explore Trade-offs

```
"What are the trade-offs between approach A and B?"
"When would I choose one over the other?"
"What are the downsides of this approach?"
```

### Practice Problems

```
"Give me a challenge to practice [concept]"
"How would I implement [feature] from scratch?"
"Walk me through building [component]"
```

### Deepen Understanding

```
"I know how to do X, but I don't understand why it works"
"Explain [concept] like I'm a junior developer"
"What should I learn next after mastering [topic]?"
```

## Learning Frameworks

### The Rule of Three

When learning something new:
1. **See it** - Watch/read an example
2. **Do it** - Implement it yourself
3. **Teach it** - Explain it to someone else

### Bloom's Taxonomy for Code

```
6. Create    → Design new solutions
5. Evaluate  → Compare approaches, make decisions
4. Analyze   → Debug, understand systems
3. Apply     → Use patterns in new contexts
2. Understand → Explain how code works
1. Remember  → Recall syntax, APIs
```

Aim to move up the levels as you learn.

### The Feynman Technique

1. Choose a concept
2. Explain it simply (no jargon)
3. Identify gaps in your explanation
4. Fill gaps and simplify further

- "What's wrong with this code?"

- "What's the best structure for this feature?"

## Mentorship Prompts

Use these to start a mentorship session:

**Understanding**
> "I'm implementing [feature]. Before I start, help me think through the architecture."

**Review**
> "Here's my code for [feature]. Review it from a learning perspective - what patterns am I missing?"

**Comparison**
> "I could use approach A or B for [problem]. Help me understand the trade-offs."

**Debugging**
> "I'm stuck on [problem]. Don't give me the answer - help me develop a debugging strategy."

**Growth**
> "I feel comfortable with [current skill]. What should I learn next to level up?"
